0. 客户需求

存算分离主键（Primary Key）表当前只能按主键 / 排序键（ORDER BY）单一维度物理聚集，存在几个缺陷：

  • 如果 PK 表指定了 ORDER BY，大 tablet 导入的很多优化无法生效。
  • 只能优化一种查询模式。
  • range bucket 不支持指定 ORDER BY —— 这会导致 tablet 分裂时需要重新拷贝大量数据。

但根据实际用户反馈，存在查询模式不匹配主键列的情况，例如：

Intuit Mailchimp

CREATE TABLE `contact_merge_fields` (
  `account_id` bigint(20) NOT NULL,
  `contact_id` varchar(256) NOT NULL,
  `merge_field_id` varchar(256) NOT NULL,
  `contact_audience_id` bigint(20) NULL,
  ... （省略其余 __header_* / __kafka_* 列）
  `__generated_partition_column_0` bigint(20) NULL AS mod(account_id, 10)
) ENGINE=OLAP
PRIMARY KEY(`account_id`, `contact_id`, `merge_field_id`)
PARTITION BY (`__generated_partition_column_0`)
DISTRIBUTED BY HASH(`contact_id`) BUCKETS 450
ORDER BY(`account_id`, `contact_audience_id`)
PROPERTIES ("enable_persistent_index"="true", "persistent_index_type"="CLOUD_NATIVE", ...);

唯一性约束由 account_id, contact_id, merge_field_id 保证，但所有查询都会涉及 account_id, contact_audience_id —— 与主键列不匹配。

Finaloop

CREATE TABLE businessEvents.businessEventActions (companyId varchar NOT NULL, id varchar ... )
PARTITION BY date_trunc('month', eventTime)
DISTRIBUTED BY HASH(`companyId`) BUCKETS 6
PROPERTIES ("enable_persistent_index"="true");

他们实际只需 companyId + eventTime + id 保证主键唯一；幸运的是 accountingMethod / accountRole 不会被修改，可加入 PK 列来加速查询。但这并非通用解。

1. StarRocks 二级索引设计

1.1 总体思路

主键表的数据在主表中按主键 / 排序键物理排列。当查询条件落在其它列上时只能全表扫描。本方案为这些列建一个轻量有序边车文件：只存「索引列 + 行位置指针」（不存整行），按索引列排序。查询时在该文件内定位命中行位置，再回主表取数。

与物理 ORDER BY 的本质区别：

  • ORDER BY 把整张表按某个键物理重排 —— 该键查询最快，但只能选一个维度，且主键点查会因主键让位而变慢。
  • 二级索引是主表之外的小文件，主表布局不变 —— 可在同一张表上为多个不同列各建一个，互不影响主键，存储开销小。

1.2 文件格式（rowset 级）

  • 粒度：每个 rowset、每个索引 → 一个独立 .idx 文件，复用 segment v2 格式。
  • Schema：[索引列 1, …, 索引列 K, __sidx_pos__ BIGINT]，按索引列排序。
  • 行位置编码：__sidx_pos__ = (segment_id << 32) | rowid_in_segment，一个 int64 唯一定位 rowset 内任意一行。
  • 加速结构：索引列默认开启 page 级 zone-map + bloom filter，等值/范围查询可页内短路。
  • 文件名：sidx_<tablet>_<txn>_<index>.idx，含 txn 天然标识数据版本。

1.3 写入路径

写入时由 in-line collector 在 TabletWriter::write() 直接从写入 chunk 抽取索引列值 + 行位置，finish() 时排序并落 .idx。全程不读回已写数据。

1.4 读取路径

查询谓词
  │  ① 前缀匹配：谓词必须命中索引「首列」，否则跳过该索引（避免无效全文件扫）
  ▼
TabletReader::get_segment_iterators
  │  ② open .idx（进程级 Reader 缓存：footer + column reader 只初始化一次）
  │  ③ lookup：谓词下推到 .idx segment scan（zone-map/bloom 剪枝）→ 解码位置
  │     → 每 segment 一个 Roaring 候选位图    ★ lookup 结果跨 morsel 共享缓存
  │  ④ 多索引：各索引候选位图按 AND 合并
  ▼
经 SparseRange 喂给 base scan 的 presupplied_rowid_filter → 只读候选行
（DelVec / 残差谓词仍由既有 scan pipeline 处理）

免回表（覆盖）查询走独立快路径：当输出列也全部在索引内时，直接从 .idx 产出结果，不经上面的回表步骤（见 §2.7 / §3.5）。

1.5 关键设计要点

设计作用
谓词前缀匹配仅当查询谓词命中索引「首列」时才启用该索引，否则跳过，保证无关查询零开销
索引列 zone-map + bloom.idx 内等值/范围查询页内短路，命中页大幅减少
Reader 缓存.idx 的 Segment（footer + column reader）进程级缓存，避免每次查询重复打开
Lookup 结果共享同一查询、同一 .idx 的候选位图只计算一次并复用，避免并行扫描下的重复查找
多索引 AND 合并同一 tablet 多个索引的候选位图按位与，进一步缩小候选集

1.6 当前 PoC 边界

  • FE 侧 DDL（CREATE INDEX）未接，索引通过 BE 配置 secondary_index_defs 注册（后门）。
  • 写入需关闭 load spill（spill 并行 merge 路径的 clone() 尚未携带 collector）。
  • 仅存算分离；ALTER ADD INDEX、CBO 选择等列入后续阶段。

2.7 免回表（覆盖）查询测试设计：排序键 vs 非排序键

目的：验证「查询不需要回表」时，二级索引相比 ORDER BY 的表现。关键区分谓词列是否为表的物理排序键。

表物理排序二级索引
orders无（靠二级索引）idx_user_town(user_id, town)、idx_ts(create_ts)
orders_obORDER BY(user_id, town)无（对照）

两类谓词：

  • 排序键谓词：WHERE user_id BETWEEN ... —— user_id 正是 orders_ob 的前导排序键（ORDER BY 的主场）。
  • 非排序键谓词：WHERE create_ts BETWEEN ... —— create_ts 不在 orders_ob 的排序键里（ORDER BY 无法剪枝，被迫全表扫）。

二级索引注册（PoC 尚无 CREATE INDEX，经 BE config 注册，需在导入前开启）：

curl -X POST "http://${CN}:8040/api/update_config?enable_secondary_index_write=true"
curl -X POST "http://${CN}:8040/api/update_config?secondary_index_defs=<tablet_id>:idx_user_town:user_id,town;<tablet_id>:idx_ts:create_ts"

测试查询（免回表 COUNT(*)，输出列 ⊆ 索引列）：

-- 排序键谓词（= orders_ob 前导排序键）
SELECT COUNT(*) FROM orders WHERE user_id BETWEEN '12345' AND '12444';   -- 0.55%  (550K 行)
SELECT COUNT(*) FROM orders WHERE user_id BETWEEN '14000' AND '14999';   -- 5.5%   (5.5M 行)
SELECT COUNT(*) FROM orders WHERE user_id BETWEEN '10000' AND '14999';   -- 27.8%  (27.8M 行)

-- 非排序键谓词（不在 orders_ob 排序键内）
SELECT COUNT(*) FROM orders WHERE create_ts BETWEEN '2025-06-01 00:00:00' AND '2025-06-02 00:00:00'; -- 1 天 (114K)
SELECT COUNT(*) FROM orders WHERE create_ts BETWEEN '2025-06-01 00:00:00' AND '2025-06-08 00:00:00'; -- 1 周 (802K)
SELECT COUNT(*) FROM orders WHERE create_ts BETWEEN '2025-06-01 00:00:00' AND '2025-07-01 00:00:00'; -- 1 月 (3.4M)

三条路径切换（同一条 SQL，只改配置 / 改表）：

路径查询的表配置
覆盖（免回表）ordersenable_secondary_index_read=true + enable_secondary_index_covering=true
lookup（回表）ordersenable_secondary_index_read=true + enable_secondary_index_covering=false
ORDER BY（对照）orders_ob—

覆盖路径的两项关键优化：

  • 并行切片：.idx 每 base 行一条记录（N_idx == N_base），据此把 .idx 行空间按 morsel 的 base-rowid 覆盖映射成不相交切片，各 morsel 并行扫各自切片，无重复扫描。
  • 无删除快路径：rowset 无 delete vector 时，跳过读取 8 字节/行的 __sidx_pos__ 位置列、跳过逐行解码与 DelVec 判定，COUNT(*) 退化为「数匹配的 .idx 行数」。

2.8 测量方法

  • 每查询 15 次取 p50/p95，预热 3 次。
  • datacache 冷/热两种模式（清 datacache + 重启 CN 制造冷态）。
  • 通过 ANALYZE PROFILE + BE /api/profile 的 SecondaryIndex 分组定位瓶颈。

结论：二级索引的价值取决于「全表扫有多贵」，基表扫描越便宜、索引能省的越少。

  • ORDER BY 不受列类型这层影响（物理聚集、连续读），两种类型下在其排序键上全程最快。

免回表场景核心结论：
  • 非排序键谓词（二级索引的核心价值场景）：覆盖索引比 ORDER BY 快 3–6×，且全选择度区间都赢、也全面赢带回表的 lookup。一张表只能物理排序一个键，但可建多个二级索引。
  • 排序键谓词（ORDER BY 本命）：差距已收窄到 ~1.3×。剩余差距为构造性下界 —— ORDER BY 做 COUNT(*) 几乎不读列数据（靠区间边界），覆盖索引至少仍需读一个 carrier 列。

4. PoC 总结

4.1 结论

  • 目标场景显著加速：最严苛（全 VARCHAR）回表场景下，≤10% 选择度查询相对全表扫 1.2–6.6× 加速，高选择性等值达 6.6×。
  • 适用边界清晰：回表场景索引收益随选择度递减，交叉点 ~10–12% —— 超过应退回全表扫（可由 CBO 自动选择）。
  • 免回表（覆盖）已落地且更强：在非排序键谓词上 比 ORDER BY 快 3–6×，在排序键上也逼近到 ~1.3× —— 这是二级索引相对 ORDER BY 最有竞争力的场景。

4.2 二级索引 vs ORDER BY 取舍

维度ORDER BY二级索引
加速维度数1 个（单一物理排序）多个（每列可独立建）
排序列查询性能最快次之（覆盖路径 ~1.3×、回表 1.1–3.6× 慢）
非排序列查询性能无法加速（全表扫）数量级加速（覆盖 3–6×）
主键点查受损（48ms）不受影响（17ms）
写入开销相比原始表高（+44%）与 ORDER BY 接近，publish 代价更小
额外存储0+索引文件（≈6.5%）

单一查询模式选 ORDER BY；多维查询模式只能用二级索引（ORDER BY 不可叠加）。
