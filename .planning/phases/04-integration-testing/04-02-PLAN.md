**test_adbc_ddl (T file):**
Test cases (each with `-- name:` marker):

1. `test_adbc_create_and_show`: CREATE EXTERNAL CATALOG with type=adbc, adbc.driver=flight_sql, adbc.url=grpc://${external_flightsql_ip}:${external_flightsql_port}. Then SHOW DATABASES, SHOW TABLES, DESCRIBE a table. Cleanup: set catalog default_catalog; DROP CATALOG.

2. `test_adbc_alter_catalog`: CREATE CATALOG, ALTER CATALOG to change a property, verify with SHOW CREATE CATALOG (if supported) or just that catalog still works. Cleanup: DROP CATALOG.

3. `test_adbc_drop_catalog`: CREATE CATALOG, verify it exists via SHOW CATALOGS, DROP CATALOG, verify it's gone.

All catalogs use `adbc_catalog_${uuid0}` naming pattern.
All test cases start with `set catalog default_catalog;` and end with cleanup.

**test_adbc_query (T file):**
Test cases:

1. `test_adbc_select_all`: SELECT * FROM test_db.test_types ORDER BY id -- returns all 5 rows
2. `test_adbc_column_pruning`: SELECT id, name FROM test_db.test_types ORDER BY id -- returns only 2 columns
3. `test_adbc_predicate_pushdown`: SELECT name FROM test_db.test_types WHERE id > 3 ORDER BY name -- returns delta, epsilon
4. `test_adbc_limit_pushdown`: SELECT name FROM test_db.test_types LIMIT 2 -- returns 2 rows (order may vary)
5. `test_adbc_combined_pushdown`: SELECT name FROM test_db.test_types WHERE flag = true ORDER BY id LIMIT 2 -- returns alpha, gamma
6. `test_adbc_explain_pushdown`: EXPLAIN SELECT name FROM test_db.test_types WHERE id > 3 -- assert contains ADBC and shows pushed SQL (use [UC] tag since EXPLAIN output varies)
7. `test_adbc_subquery`: SELECT name FROM test_db.test_types WHERE id IN (SELECT id FROM test_db.test_join WHERE category = 'A') ORDER BY name -- returns alpha, gamma

Each query test case creates catalog at start and drops at end. Use the same catalog for all queries within a test case to reduce overhead. Reference tables from seed data: test_db.test_types, test_db.test_join.

**test_adbc_mv (T file):**
Test cases:

1. `test_adbc_mv_lifecycle`: Create catalog, create a native StarRocks database + table, CREATE MATERIALIZED VIEW mv_${uuid0} AS SELECT id, name FROM adbc_catalog.test_db.test_types, REFRESH MATERIALIZED VIEW mv_${uuid0}, wait for refresh, SELECT from MV, DROP MV, cleanup.

2. `test_adbc_mv_query_rewrite`: Create MV, refresh it, then EXPLAIN a query that should rewrite to use the MV (use [UC] tag for EXPLAIN).

MV tests need a native StarRocks database to store the MV in. Create `db_mv_${uuid0}` for this. Clean up everything at end.

**R files:**
For initial creation, write R files with empty result blocks since actual results depend on a running Flight SQL server. Mark results with placeholder comments. The R files will be properly generated when running in record mode (`python3 run.py -r`) against a real server.

For EXPLAIN and non-deterministic output, use `[UC]` tag (unchecked) so results aren't validated.

IMPORTANT formatting rules from test/CLAUDE.md:

- Use `${external_flightsql_ip}` and `${external_flightsql_port}` from sr.conf
- Always `set catalog default_catalog;` before cleanup
- R file results use `-- result:` / `-- !result` markers
